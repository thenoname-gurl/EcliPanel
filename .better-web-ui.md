# Design Context

## Users

EcliPanel serves a **mixed audience** spanning three core user groups:

- **Game server owners / managers** — running game servers for communities or clients. They need reliable uptime, server lifecycle management, and multi-node orchestration.
- **Developers deploying apps** — deploying web applications and backends. They need fast provisioning, DNS management, and CI/CD-friendly workflows.
- **Hosting providers / teams** — reselling hosting services or managing infrastructure for an organization. They need multi-tenant orgs, billing, ticketing, and admin tooling.

**Job to be done**: Provision, manage, monitor, and secure game servers and applications across a distributed node network — from a single panel.

**Desired emotions**: The interface should simultaneously evoke **confidence & trust**, **speed & responsiveness**, **delight & polish**, and **clarity & simplicity**.

## Brand Personality

- **Tone**: Modern, Clean, Professional — with a **youth/developer-focused** edge.
- **3-word archetype**: *Modern, Clean, Professional* with a developer-first sensibility.
- **Tone range**: Balanced middle ground — professional but with personality. Not overly corporate, not overly playful.
- **Language**: Neutral-to-conversational. Technical where appropriate, welcoming where it counts.
- **Emotional goals**: Inspire confidence in the platform's reliability while making complex infrastructure feel approachable.

## Aesthetic Direction

- **Visual tone**: Dark-first with light theme options. Current direction is the established baseline — purple primary (`#8b5cf6`), deep backgrounds (`#0a0a12`), subtle glow effects.
- **14 themes** defined in `frontend/lib/themes.ts` capture the full visual range: Eclipse Purple (default), Cyber Blue, Neon Green, Solar Orange, Ruby Red, Voters 7 Mystery, Gambling Mode Dark, Arctic White, Arctic Snow, Frost Beam, Nordic Light, Bubblegum Pink, Gambling Mode White — each with complete CSS variable sets for primary, bg, card, secondary, sidebar, accent, glow, border, foreground, etc.
- **Corner shape**: `0.75rem` radius (12px) — gently rounded, leaning rounded but controlled.
- **Palette lean**: Default purple palette leans slightly cool (purple/violet primary on dark). Light themes lean warm or neutral.
- **Fonts**: Geist (sans), Geist Mono (monospace), Didact Gothic (display/headings used via `font-flink` and `font-heading`).
- **Glow system**: Custom `glow-box`, `glow-box-sm`, `glow-text`, `glow-border` CSS utilities using semi-transparent primary color shadows. Used sparingly for emphasis.
- **Animation**: framer-motion for micro-interactions and page transitions. Custom view-transition with a Shigure scale wipe animation for theme switching. Custom keyframes: `pulse-glow`, `float`, `rotate-slow`.
- **Anti-references**: Not explicitly stated. The existing aesthetic avoids skeuomorphism and overly playful "gaming" clichés despite the game server audience.

## shadcn Customizations

- **Aliases**: `@/components/ui` for UI components, `@/lib/utils` for `cn()`, `@/hooks` for hooks. Standard shadcn layout.
- **Theme engine**: Overrides shadcn's default CSS variables per-theme via inline script in `layout.tsx` that injects from `THEMES` array. The `globals.css` defines the default (Eclipse Purple) as `:root` (no light mode distinction — all themes are explicit).
- **Sidebar**: Custom shadcn sidebar implementation in `@/components/ui/sidebar.tsx` (~726 lines) with cookie-persisted state, mobile sheet, keyboard shortcut, tooltips.
- **Custom additions beyond stock shadcn**: `field.tsx`, `item.tsx`, `button-group.tsx`, `input-group.tsx`, `kbd.tsx`, `empty.tsx`, `spinner.tsx`, `chart.tsx` (recharts wrapper).
- **Not used**: No TanStack Table, no TanStack Form, no TanStack Virtual. No shadcn Blocks.

## Design Principles

1. **Establish hierarchy with spacing, weight, and typography before adding color or glow.** The dark theme provides structural contrast; avoid relying on color alone to differentiate elements.

2. **Use glow and animation sparingly as emphasis, not decoration.** The existing glow system (`glow-box`, `glow-text`) exists for focus states and primary actions — not for surface-level ornament.

3. **Prioritize speed and clarity in every interaction.** Users managing infrastructure need to act fast. Minimize clicks, keep loading states tight, and make system status immediately scannable.

4. **Keep one obvious primary action per context.** Whether it's starting a server, creating a ticket, or applying a plan — make the primary action visually unambiguous without competing calls-to-action.

5. **Honor the theme system.** All 14 themes must work visually without breakage. Never hardcode color values — always use CSS variables (`--primary`, `--background`, etc.) and test across dark and light themes.

6. **Design for a mixed audience without alienating any group.** UI should be technical enough for developers, intuitive enough for game server owners, and structured enough for hosting providers.
