# Design Context — EcliPanel v3

> Canonical reference for the frontend design system. Updated 2026-07-09 against the live codebase.

---

## 1. Users

EcliPanel serves a **mixed audience** spanning three core user groups:

- **Game server owners / managers** — running game servers for communities or clients. They need reliable uptime, server lifecycle management, and multi-node orchestration.
- **Developers deploying apps** — deploying web applications and backends. They need fast provisioning, DNS management, and CI/CD-friendly workflows.
- **Hosting providers / teams** — reselling hosting services or managing infrastructure for an organization. They need multi-tenant orgs, billing, ticketing, and admin tooling.

**Job to be done**: Provision, manage, monitor, and secure game servers and applications across a distributed node network — from a single panel.

**Desired emotions**: The interface should simultaneously evoke **confidence & trust**, **speed & responsiveness**, **delight & polish**, and **clarity & simplicity**.

---

## 2. Brand Personality

- **Tone**: Modern, Clean, Professional — with a **youth/developer-focused** edge.
- **3-word archetype**: *Modern, Clean, Professional* with a developer-first sensibility.
- **Tone range**: Balanced middle ground — professional but with personality. Not overly corporate, not overly playful.
- **Language**: Neutral-to-conversational. Technical where appropriate, welcoming where it counts.
- **Emotional goals**: Inspire confidence in the platform's reliability while making complex infrastructure feel approachable.

## 3. Two Visual Systems

The frontend uses **two distinct visual approaches** depending on context:

### 3.1 Dashboard (Themable System)

The core dashboard (`/dashboard/*`) is built on a **fully themable CSS variable system** powered by shadcn/ui semantics. Every component references CSS custom properties — never hardcoded colors — so switching themes updates the entire UI instantly.

- **14 themes** defined in `frontend/lib/themes.ts`
- Theme is applied by an inline `<script>` in `layout.tsx` that sets CSS variables on `:root` before first paint (prevents FOUC)
- Theme preference persisted in localStorage + user settings API
- Animated theme switching via the View Transitions API with a Shigure scale-wipe mask animation
- All shadcn/ui components (`button`, `card`, `input`, `dialog`, `badge`, etc.) respect the active theme automatically

### 3.2 Landing / Marketing / V2 Server Pages (Hardcoded System)

Landing pages (`/landing/*`), docs pages, the ELO showcase, and Proxmox server detail pages (`ServerViewV2`) use a **fixed dark aesthetic** with hardcoded colors. These pages are deliberately distinct from the dashboard — they follow a landing/docs design language:

- Background: `#0a0a0f` or `#050507` (near-black)
- Text: `text-white` with `text-white/60`–`text-white/80` for secondary
- Cards: `border border-white/20` or `border-white/10`
- Accent: purple `#8b5cf6`, cyan `#06b6d4`, green `#10b981`, amber `#f59e0b`
- Fonts: `font-flink` (Didact Gothic) for headings/numbers, `font-inter` for body
- Rich animated backgrounds (PixelBlast, GradualBlur, ShinyText from React Bits)
- Heavy framer-motion micro-interactions

**Rule of thumb**: Dashboard = theme variables. Landing/V2 = hardcoded dark aesthetic.

## 4. Theme System

### 4.1 CSS Variable Architecture

All theme colors flow through a two-layer system:

**Layer 1 — shadcn semantic tokens** (defined in `globals.css` `:root`):
```
--background, --foreground, --card, --card-foreground, --popover, --popover-foreground,
--primary, --primary-foreground, --secondary, --secondary-foreground,
--muted, --muted-foreground, --accent, --accent-foreground,
--destructive, --destructive-foreground, --border, --input, --ring
```

**Layer 2 — EcliPanel extended tokens** (also in `:root`):
```
--chart-1 → --chart-5    (recharts color palette)
--glow, --glow-strong     (box-shadow glow effects)
--success, --warning, --info  (semantic status colors)
--sidebar, --sidebar-foreground, --sidebar-primary, --sidebar-primary-foreground,
--sidebar-accent, --sidebar-accent-foreground, --sidebar-border, --sidebar-ring
--radius: 0.75rem          (corner radius, with derived --radius-sm/md/lg/xl)
```

**Layer 3 — Tailwind v4 `@theme inline`** maps every CSS variable to a Tailwind utility:
```css
--color-background: var(--background);
--color-primary: var(--primary);
--color-glow: var(--glow);
/* ... all tokens mapped */
```
This means you write `bg-background`, `text-primary`, `border-border`, `shadow-[var(--glow)]` and they resolve to the active theme's values.

## 5. Typography

### 5.1 Font Stack

| Token | Font | Usage |
|-------|------|-------|
| `--font-sans` / `font-sans` | **Geist** (via `next/font/google`) | Body text, UI labels, general content |
| `--font-mono` / `font-mono` | **Geist Mono** | Code, terminal output, monospace contexts |

| `--font-heading` / `font-heading` | **Didact Gothic** (alias) | Same as font-flink, used in landing/dashboard headings |

**Loading**: All fonts loaded via `next/font/google` in `layout.tsx` with `display: "swap"` and `preconnect` hints.

**Tailwind config**: Fonts are registered as `font-sans`, `font-mono`, `font-inter` (maps to Geist), `font-flink`, `font-heading`.

### 5.2 Typography Scale

Uses shadcn/new-york defaults:
- Body: `text-sm` (0.875rem), `text-base` (1rem)
- Headings: `text-lg` through `text-7xl` with `font-semibold` or `font-bold`
- Captions/labels: `text-xs` with `tracking-widest uppercase` for stat labels
- Monospace: `text-sm` in code contexts
- Display/hero: `text-5xl`–`text-[6.4rem]` with `font-heading` on landing

## 7. Spatial Design

### 7.1 Corner Radius

- Base: `--radius: 0.75rem` (12px) — gently rounded
- Derived: `--radius-sm: calc(var(--radius) - 4px)`, `--radius-md: calc(var(--radius) - 2px)`, `--radius-lg: var(--radius)`, `--radius-xl: calc(var(--radius) + 4px)`
- Cards: sharp (no radius class), relying on parent container radius
- Buttons: inherit from theme
- Modals/dialogs: sharp by default, rounded via `--radius`

### 7.2 Spacing & Layout

- Dashboard shell: `flex min-h-screen` with collapsible sidebar + header + main content
- Header: `h-14 sm:h-16` with `border-b border-border`
- Content padding: `px-3 sm:px-6` (responsive)
- Cards: `p-6` internally, `gap-6` in flex column layout
- Section gaps: `gap-2` to `gap-4` for form/inline, `gap-6` for card grids

### 7.3 Surface Hierarchy

```
Background (--background)
  └─ Card (--card) with border (--border) and shadow-sm
       └─ Card header/content/footer sections
  └─ Sidebar (--sidebar) with its own color tokens
  └─ Header bar: bg-card/50 + backdrop-blur-sm (semi-transparent glass)
  └─ Modal overlays: bg-black/50 or bg-background/80 + backdrop-blur-sm
```

### 7.4 Glass / Backdrop Blur Effects

Used extensively for overlays and semi-transparent surfaces:
- Sidebar modals: `bg-black/60 backdrop-blur-sm`
- Header: `bg-card/50 backdrop-blur-sm`
- KYC banner overlay: `bg-background/95 backdrop-blur-md`
- Command palette overlay: `bg-background/80 backdrop-blur-sm`
- General overlay pattern: `fixed inset-0 bg-black/50 backdrop-blur-sm`

| `--border` | All borders (cards, inputs, separators) |
| `--input` | Input field borders |
| `--ring` | Focus rings, matches primary |

### 8.2 Glow System

Three CSS utility classes for emphasis effects (used sparingly):

```css
.glow-box      → box-shadow: 0 0 20px var(--glow), 0 0 60px var(--glow), inset 0 0 20px rgba(139,92,246,0.05)
.glow-box-sm   → box-shadow: 0 0 10px var(--glow)
.glow-text     → text-shadow: 0 0 10px var(--glow)
.glow-border   → box-shadow: 0 0 8px var(--glow), inset 0 0 8px rgba(139,92,246,0.03)
```

The `--glow` value is theme-specific (e.g., `#8b5cf680` for Eclipse Purple, `#9CA3AF40` for Voters 7). The inset glow is always purple-tinted as a deliberate brand accent.

### 8.3 Chart Colors

`--chart-1` through `--chart-5` are used by the recharts wrapper (`chart.tsx`):
- chart-1 = primary, chart-2 = cyan, chart-3 = green, chart-4 = amber, chart-5 = red

### 8.4 Dark/Light

All 14 themes handle both modes by defining complete variable sets. There is no automatic light/dark switching — themes are selected explicitly. The `:root` block serves as the default theme (currently Voters 7 Mystery, dark). The `.dark` class mirrors `:root` for compatibility but the primary theme mechanism is the inline script injection in `layout.tsx`.

## 9. Animation & Motion

### 9.1 framer-motion

Used for:
- Page/section entrance animations (`initial`, `animate` with opacity/translateY)
- Stagger children in feature grids
- Hover scale effects on interactive cards
- Hero text reveals on landing pages
- Loading skeletons and transitions

Typical pattern:
```tsx
<motion.div
  initial={{ opacity: 0, y: 20 }}
  animate={{ opacity: 1, y: 0 }}
  transition={{ duration: 0.5, ease: "easeOut" }}
>
```

### 9.2 CSS Keyframe Animations

Defined in `globals.css`:

| Animation | Duration | Behavior |
|-----------|----------|----------|
| `pulse-glow` | 2s | Opacity pulse (0.6 ↔ 1.0), used on status indicators |
| `float` | 3s | Gentle Y-axis float (-4px), used on decorative elements |
| `rotate-slow` | 40s | Full rotation, used on background ornaments |
| `scale` | 2.5s | Mask-size wipe for theme transitions (0 → 2000vmax) |

Utility classes: `.animate-pulse-glow`, `.animate-float`, `.animate-rotate-slow`.

### 9.3 View Transitions (Theme Switching)

The Shigure scale-wipe uses `::view-transition-new(root)` and `::view-transition-old(root)` pseudo-elements with a `scale` keyframe on `mask-size`. Duration controlled by `--switch-duration` (default 2.5s, set via inline style). The mask image is `/assets/shigure-ui-smol.gif`. Falls back to instant for `prefers-reduced-motion`.

### 9.4 Scroll Behavior

`html { scroll-behavior: smooth; }` — all anchor links and scroll actions are smooth by default.

## 10. Interactive Element Patterns

### 10.1 Focus States

All interactive elements use `focus-visible:ring-ring/50 focus-visible:ring-[3px]` with `outline-none`. This creates a visible focus ring using the theme's primary color.

### 10.2 Hover States

- Buttons: `hover:bg-primary/90` (default), `hover:bg-accent` (ghost)
- Cards: `hover:shadow-md transition-shadow`
- Links: `hover:underline` or color shift
- Sidebar items: `hover:bg-accent hover:text-accent-foreground`

### 10.3 Active/Press States

- Buttons: `active:scale-95` (subtle press effect)
- Transitions: `transition-all` or `transition-colors` on most interactive elements
- Duration: typically `duration-200`

### 10.4 Disabled States

- `disabled:pointer-events-none disabled:opacity-50`
- `disabled:cursor-not-allowed`

### 10.5 Form Validation

Invalid states: `aria-invalid:ring-destructive/20 aria-invalid:border-destructive`

## 14. shadcn Customizations

- **Aliases**: `@/components/ui` (UI components), `@/lib/utils` (`cn()`), `@/hooks` (React hooks)
- **Theme engine**: Overrides shadcn default CSS variables via inline `<script>` that injects from the `THEMES` array. Default theme in `:root` is Voters 7 Mystery (grayscale dark). The `.dark` class mirrors `:root` for backward compatibility.
- **Sidebar**: Custom ~726-line implementation (`components/ui/sidebar.tsx`) — not the stock shadcn sidebar. Cookie-persisted collapsed state, mobile sheet with overlay, keyboard shortcut (Ctrl+B), tooltip labels when collapsed, organization switcher.
- **Extended shadcn components beyond stock**: `field.tsx`, `item.tsx`, `button-group.tsx`, `input-group.tsx`, `kbd.tsx`, `empty.tsx`, `spinner.tsx`, `chart.tsx` (recharts wrapper)
- **Tailwind v4 plugin**: `tw-animate-css` for extended animation utilities
- **Typography plugin**: `@tailwindcss/typography` for `.prose` content (docs, legal pages)
- **Registries**: `@react-bits` (React Bits — for PixelBlast, GradualBlur, ShinyText on landing)
- **Not used**: No TanStack Table, no TanStack Form, no TanStack Virtual, no shadcn Blocks

## 16. Design Principles

1. **Establish hierarchy with spacing, weight, and typography before adding color or glow.** The dark theme provides structural contrast; avoid relying on color alone to differentiate elements.

2. **Use glow and animation sparingly as emphasis, not decoration.** The glow system (`glow-box`, `glow-text`) exists for focus states and primary actions — not for surface-level ornament.

3. **Prioritize speed and clarity in every interaction.** Users managing infrastructure need to act fast. Minimize clicks, keep loading states tight, and make system status immediately scannable.

4. **Keep one obvious primary action per context.** Whether it's starting a server, creating a ticket, or applying a plan — make the primary action visually unambiguous without competing calls-to-action.

5. **Honor the theme system.** All 14 themes must work visually without breakage. In the dashboard, never hardcode color values — always use CSS variables (`--primary`, `--background`, etc.) or Tailwind semantic classes (`bg-background`, `text-primary`). Test across dark and light themes. **Exception**: Landing pages, docs, and ServerViewV2 intentionally use a fixed dark aesthetic with hardcoded colors — they are not theme-aware.

6. **Design for a mixed audience without alienating any group.** UI should be technical enough for developers, intuitive enough for game server owners, and structured enough for hosting providers.

7. **Two visual systems, one brand.** Landing/marketing pages are deliberately flashy (animated backgrounds, custom colors) to attract. Dashboard is deliberately restrained (themeable, consistent) for daily use. Both share the same brand personality, fonts, and purple accent color.
